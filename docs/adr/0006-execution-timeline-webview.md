# 6. Execution timeline as separate webview panel

**Date:** 2026-03-28
**Status:** Accepted

## Context

v0.4 requires a Gantt-style execution timeline showing phase start times, durations, and current progress. Two approaches were considered:

1. Extend the existing dependency graph webview to include timeline data
2. Create a separate webview panel dedicated to the timeline

The dependency graph renders a DAG of phase relationships (spatial layout). The timeline renders phase execution over time (temporal layout). These are fundamentally different visualizations with different axes, interaction models, and update frequencies.

## Decision

Create a dedicated `ExecutionTimelinePanel` webview, separate from the dependency graph.

**Rendering approach:** Inline HTML/CSS/JS generated by `renderTimelineHtml()`, consistent with the DAG pattern established in [ADR 0002](0002-webview-dag-rendering.md). No external dependencies — the HTML is self-contained with inline styles and scripts.

**Data pipeline:** `ProgressState` → `computeTimeline()` (pure function in `parsers/timeline.ts`) → `TimelineData` → `renderTimelineHtml()` → webview HTML string.

**Timestamp handling:** Phase timestamps from PROGRESS.md are treated as local time. No timezone conversion is applied. This is a simplifying assumption — claudeloop writes timestamps in the local timezone of the machine running it, and Oxveil displays on the same machine.

## Consequences

**Positive:**
- Each webview has a single responsibility — DAG shows relationships, timeline shows execution over time
- Independent update cycles — timeline can refresh on a timer (for the "NOW" indicator) without affecting the DAG
- The `computeTimeline()` parser is a pure function, fully testable with Vitest
- Consistent with the inline HTML pattern from ADR 0002 — no new rendering dependencies

**Negative:**
- Two webview panels to manage instead of one combined view
- Users must open the timeline separately (via `oxveil.showTimeline` command)
- Local time assumption breaks if claudeloop is run remotely or across timezones — acceptable for current single-machine usage
